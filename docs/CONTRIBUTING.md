# 贡献指南

本文档说明在当前仓库里提交改动时，推荐遵守的协作方式。

## 1. 基本原则

1. 先以当前源码为准，不要仅根据历史文档推断现状。
2. 变更运行链时，优先确认真正的主路径，而不是沿着 compatibility 命名继续扩写。
3. 提交“删除旧逻辑”类改动时，不要只改局部；要做 repo-wide 搜索和复核。
4. 在没有完成全仓核查前，不要提前宣称“已经删干净”。

## 2. 开发前建议

开始动手前，优先读：

- `README.md`
- `CLAUDE.md`
- `docs/ARCHITECTURE_ANALYSIS.md`
- `docs/API_DOCUMENTATION.md`

如果你的改动涉及宿主消息读取、执行链或写回：

- `modules/tool-trigger.js`
- `modules/tool-output-service.js`
- `modules/context-injector.js`

## 3. 修改不同类型内容时应同步的文件

如果你修改的是：

### 3.1 架构边界或主路径结论
同步：
- `docs/ARCHITECTURE_ANALYSIS.md`
- `README.md`
- `CLAUDE.md`（如协作指引也因此变化）

### 3.2 工具模型 / 执行行为 / 对外 API
同步：
- `docs/API_DOCUMENTATION.md`
- 相关源码注释

### 3.3 宿主验证方式
同步：
- `docs/HOST_REGRESSION_CHECKLIST.md`

### 3.4 仅为历史记录或版本说明
同步：
- `docs/CHANGELOG.md`

## 4. 关于旧自动触发相关历史资料

仓库中仍可能保留一些历史文档，用于记录：
- 当时的问题背景
- 已做过的尝试
- 宿主验收模板
- 旧方案分析

它们不一定代表当前代码仍然这样工作。

若文档与源码冲突：
- 以源码为准
- 再回头修正文档

## 5. 提交前检查建议

至少完成：

1. `npm run build`
2. 对受影响目录做一次关键词搜索
3. 如果是删除旧实现，确认：
   - 运行逻辑是否真的不可达
   - 配置 schema 是否还在写旧字段
   - UI 文案是否仍在制造错误认知
   - 文档是否仍把旧实现当成当前事实

## 6. 文档清理建议

当你确定某份文档已经主要描述历史方案而不再适合作为当前事实时，有两种做法：

1. 明确改写成“历史档案 / 背景文档”
2. 或直接删除并在其它文档中移除引用

不要让历史方案继续伪装成当前主线说明。
